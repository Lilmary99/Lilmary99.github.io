Milestone 3

Part A:
After completing Milestone 2, our group came to the conclusion that our calendar merger should have a convenient, user-friendly experience. We want our users to be able to merge calendars from various email services into one platform that has an appealing look and an easy-to-follow interface. So, our calendar merger should have standard rows and columns in imitation of the monthly calendar with color coded reminders correlating to the email service from which the reminder came. With these requirements, we developed 3 user stories:

  1. As a user, I want to sign in to each of my email accounts so that I can view all of my reminders in one place
  2. As a user, I want to be able to view scheduling conflicts so that I am able to resolve scheduling issues in one place
  3. As a user, I want be able to choose a color for each email service so that I am able to easily identify relevant reminders

These user stories are all important, but the most important ones are definitely that last 2. We wanted our implementation to focus on a user-friendly interface. The color coded reminders allow users to easily identify which reminders are from which email. Resolving scheduling conflicts in one platform allows the user to see conflicts at first glance and edit their schedules without moving across multiple platforms. Hopefully, users will find our implementation as favorable and simple.

Part B:
Since the first user story talks about enabling users to add an email account to the calendar merger, the first wireframe includes a sign-in pop up box that prompts users to enter their credentials. Additionally, the second wireframe prompts users to enter their email and password in the top right corner of the screen in order to add that email account’s calendar to the merger.

The second user story infers that users should be able to view events from both of their calendars and confirm whether there is a scheduling conflict, so the third wireframe shows a visual of two different events from two separate email accounts overlapping each other. This overlap signifies a scheduling conflict. The fourth wireframe uses a red block to signify that there is a scheduling conflict at that time block. Realistically, the user should be able to click on that red block and see the details of the scheduling conflict.

The last user story wants to be able to customize the color of the events that show up in the merger, according to that event’s respective email account, so the fifth wireframe provides the user with a settings pop-up box which allows users to select their color preferences. The sixth wireframe has the same settings pop-up box but it uses elements from the second wireframe that includes the sign-in fields in the top right corner.

Part C:
After reviewing our proposed wireframes, we sought to create mockups that inherited the most favorable attributes from each individual wireframe. To begin with, we thought it was necessary to include the sign in module in the center of the first mockup from the initial wireframe so that users are able to sign in to a parent account. That way, they could still have access to all the children accounts. We also added the option for users to insert their profile picture for a more personalized feel, which was not included in the initial wireframe. The other option, signing into each account individually, does not provide as much security for the user. Otherwise, essentially anyone would be able to open and view the calendar if they had the user’s device. For the second wireframe, we decided to give users the option to see both of the scheduling events if there is a time conflict. The other option, in which the user would just see a red block to designate a time conflict, would not allow the user to see the actual details of said conflicting events, like the event name and time. In the last four mockups, we chose to not give the option to add an email account in the top right-hand corner of the screen. Because this is the actual page where users would go in and manually add a new account, it seemed rather redundant. Instead, we sought to include a button giving users the option to return to the umbrella calendar if they chose to.

While creating the mockups we found that it was inevitable that we implement more design decisions than we initially did for the wireframes. We wanted to include the drop-down options for colors and email accounts to get a better feel of what it would look like in terms of functionality. Ideally, once the user clicks the down caret icon, they will be able to view all of the color options and email account options, respectively. Each color name followed a colored circle representing the exact shade so that users can easily identify the color they chose. After submitting, the screen would return to the homepage where the calendar resides, and the user would be able to see that the color for the respective email account has been configured.
